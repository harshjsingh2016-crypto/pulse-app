01 Purpose & Concept

Pulse is a personal life-management app built around a single idea: everything you track daily should be accessible through one conversation. You tell Claude what you did, what you ate, or what you spent — and it logs it, asks the right follow-up, and confirms before writing anything. The structured screens (tasks, meals, spends) exist to review, edit, and get an overview — but adding and updating happens naturally in chat.

The app answers three questions depending on where you are:

ScreenQuestion it answersPrimary mode
ChatWhat do I need to tell Claude right now?Input & AI interaction
TasksWhat do I need to do today, and am I covering all my domains?Review & drag-reorder
MealsWhat have I eaten today and how does it track against my goals?Review & manual edit
SpendsWhere is my money going and am I within my budgets?Review & category breakdown
RecurringHave I done my standing commitments today?Quick check-off
📓
Relationship to WMS: Pulse is a sibling prototype — same owner, different persona. Where WMS is a medieval counsellor for life intent and habits, Pulse is a warm, personal notebook for daily operational life. The two may converge in a later phase; for now they are separate apps with shared architectural thinking.
02 Design Principles
PrincipleWhat it means in practice
Chat is the input layer Adding and modifying data happens in chat. Structured screens are for overview and review, not primary data entry. Claude is the form.
Propose, don't execute When Claude interprets a message as a data action (add task, log meal, record spend), it shows a confirmation card before writing to Firestore. The user always has the final say.
One context window, full picture Every chat turn includes a live snapshot of all three modules (tasks, today's meals, today's spends). Claude never reasons from stale data. No module switching — Claude figures out what you're talking about.
Work and personal live together Tasks are grouped visually into Work and Personal sections — same list, same priority buckets, same Claude context. No context switching required.
Recurring tasks are structurally separate Recurring habits live in their own Firestore collection, on their own screen, with insert/delete completion model. History is never lost.
Claude infers macros — user corrects Meal logging is free-text. Claude returns structured calorie/macro estimates. The user sees the estimate and can correct any value before it's saved. No food database lookup — Claude handles Indian meals, mixed dishes, and home cooking natively.
Budgets are per-category, per-period Each spend category has its own budget period (monthly or weekly). Progress is shown as a bar, not a raw number. Overspend is surfaced visually, not buried in a table.
Security at the database level Firestore Security Rules enforce per-user data isolation. Application code does not re-implement permission checks. A bug in a Cloud Function cannot leak another user's data.
03 Tech Stack & Rationale
LayerChoiceRationale
Mobile App React Native (Expo) Native feel on iOS and Android — smooth gestures, haptics, proper swipe actions. Expo Web gives a functional laptop experience from the same codebase. Single repo, two targets.
Web (laptop) Expo Web Runs the same React Native code in a browser via react-native-web. Sidebar nav replaces bottom tab bar on wide screens. No second codebase needed.
Auth Firebase Auth — Email OTP No passwords. sendSignInLinkToEmail sends a magic link; user taps it to authenticate. Multi-user from day one. Works on mobile and web with the same SDK.
Database Cloud Firestore Document model suits the denormalised shape of this app (tasks with subtasks, meal entries, spend entries). Real-time listeners mean the task list updates live when Claude confirms an action. Security Rules enforce per-user isolation at the DB layer.
Backend / API Firebase Cloud Functions (Node.js) All Anthropic API calls happen server-side. The ANTHROPIC_API_KEY never reaches the client. Functions also handle budget recalculation and recurring task seeding on schedule.
AI Anthropic claude-sonnet-4-6 Handles intent classification, macro inference, action proposal generation, and conversational response in one call. Best price/quality ratio for this pattern.
Hosting Firebase Hosting (web) + Expo EAS (mobile) Firebase Hosting deploys the Expo Web build with zero config. EAS Build handles iOS and Android native builds.
Local dev VS Code + Firebase Emulator Suite Firestore, Auth, and Functions all emulate locally. No cloud reads/writes during development. firebase emulators:start is the only command needed.

04 Visual Identity

Pulse uses a warm notebook aesthetic — the feeling of a well-used personal planner, not a corporate productivity tool. The palette draws from aged paper, ink, and leather. Typography mixes a grounded serif for headings with a clean sans for body text and a mono face for data.

Typography

RoleTypefaceUsage
Display / HeadingsLora (serif)Screen titles, section headers, the app name. Warm and literary without feeling formal.
Body / UIInterTask titles, chat messages, labels, buttons. Clean and highly legible at small sizes.
Data / MonoJetBrains MonoCalorie numbers, budget amounts, dates, reference codes. Tabular figures align cleanly.

Signature Element

The single memorable visual element: ruled lines as section dividers — thin horizontal lines that look like notebook paper, used to separate Work from Personal in the task list, separate meal entries by date, and separate budget categories. The lines use the Ruled Line colour and appear at 1px with a slight opacity fade at the edges. This is not decoration — it literally evokes the notebook the app is named after.

Dark Mode

Dark mode inverts to an ink-on-dark-paper palette: background #1C1612, surface #2A2018, text #F0E8D8, accent #C4956A. The ruled-line motif becomes visible as subtle lighter lines on the dark surface. Dark mode is the default for the mobile app; light mode is the default for web (laptop).

5.1 Chat Screen

The default landing screen. A full-width conversation surface. Claude has full context of all three modules on every turn — tasks (all active, including subtasks and domain), today's meal entries with running macro totals, and today's spends with budget consumption per category.

Chat behaviours

User saysClaude doesRenders
"Add call Priya to my work tasks for today"Infers: create task, domain=Work, bucket=todayAction confirmation card
"I had dal rice for lunch, maybe 400g"Estimates calories + macros from free textMeal confirmation card with editable values
"Spent 850 on groceries"Infers category=Groceries, amount=850Spend confirmation card
"What's my spend looking like this month?"Summarises category totals vs budgets from contextText response with inline numbers
"Move the vendor call to tomorrow"Finds the task, proposes bucket updateAction confirmation card
"Am I hitting my protein today?"Reads today's meal context, gives assessmentText response

Action confirmation card

When Claude proposes a data write, it emits a structured card below its message bubble. The card shows:

  • Action type badge (ADD TASK / LOG MEAL / RECORD SPEND / UPDATE / DELETE)
  • The proposed data fields in a compact summary
  • For meals: editable calorie/macro fields inline in the card before confirming
  • A Confirm button and a Cancel button
  • Confirming calls the relevant Cloud Function which writes to Firestore
⚠️
Claude never writes to Firestore directly from the client. The client sends the user message to a Cloud Function. The Cloud Function calls the Anthropic API, parses the structured action response, returns it to the client as a proposed action. The client renders the confirmation card. On user confirmation, the client calls a second Cloud Function to execute the write. Two-step, always.

Context snapshot sent on every turn

5.2 Tasks Screen

The daily task management surface. Two view modes toggled by a segmented control at the top: Priority view (default) and Domain view. Within each view, tasks are grouped into Work and Personal sections separated by a ruled line.

Priority view

  • Critical tasks pinned at the top of their group (Work critical / Personal critical) with a vermilion flame badge
  • Below: Today / Tomorrow / Later as collapsible labelled sections within each group
  • Manual drag-to-reorder within each bucket (updates priority_rank)
  • Reference date shown as a small chip — display only, never drives order

Domain view

  • One collapsible section per domain, sorted by domain sort_order
  • Work domains appear first; Personal domains below, separated by a ruled line
  • Critical tasks appear inline within their domain with a flame badge — not pulled into a separate section

Task card

  • Title, domain colour chip, date chip (if set), subtask progress chip (e.g. 2/4 sub)
  • Work/Personal label as a small pill on the card
  • Tap title → opens Task Drawer
  • Flame icon → toggles is_critical directly from the list
  • Swipe left → delete

Task Drawer (slide-up sheet)

  • Edit: title, critical toggle, reference date, domain, workspace (work/personal), notes
  • Subtasks: add / check off / delete inline
  • Per-task Claude chat: conversation scoped to this single task only. Context: task title, notes, date, critical flag, subtask state.
  • Delete task button

5.3 Meals Screen

Daily nutrition overview. The screen defaults to today's date; a date strip at the top allows scrolling back to past days.

Daily summary bar

  • Horizontal progress bars for Calories, Protein, Carbs, Fat against a user-set daily goal
  • Numbers shown in JetBrains Mono: 1240 / 2000 kcal
  • Bar colour: sage green below goal, amber near goal, vermilion over

Meal entries list

  • Grouped by meal type: Breakfast / Lunch / Dinner / Snack
  • Each entry: description, cal, P/C/F in small mono text
  • Tap entry → edit sheet (all fields editable, including Claude's macro estimates)
  • Swipe left → delete
  • Quick-add button (+) opens a text input — same flow as chat, but scoped to meal logging only

Macro goals

Set once in Profile settings. Used to drive the progress bars. Claude references these goals when answering nutrition questions in chat.

5.4 Spends Screen

Budget tracker and spend log. Defaults to the current month. A period selector (week / month) at the top switches between the two views — weekly budgets show the current week, monthly budgets show the current month.

Budget overview

  • Card per category showing: category name, budget period chip (Weekly / Monthly), amount spent vs budget as a progress bar, remaining amount in mono text
  • Overspent categories show in vermilion with an overrun amount
  • Tap category card → drill down to entries list for that category

Spend entries

  • Chronological list within a category: amount, note, date
  • Tap entry → edit sheet
  • Swipe left → delete

Category management

  • Categories are user-defined (e.g. Groceries, Eating Out, Travel, Subscriptions)
  • Each category has a name, colour, budget amount, and budget period (Weekly or Monthly)
  • Managed from a Settings screen (accessible from Spends header)

5.5 Recurring Screen

Standing commitments — habits and daily routines. Different visual language from Tasks: less "list to clear," more "rhythm to sustain."

  • Daily checklist, grouped by domain
  • Large tap targets for completion toggle
  • 7-day dot strip per item (filled = completed, empty = missed)
  • Streak count displayed per item in mono text
  • Marking done = insert a completion document for today's date. Undo = delete that document.
  • No drag reorder, no priority buckets, no critical flag
  • Tap item → notes visible in a small expand; no separate drawer
06 Firestore Data Model
📐
Firestore is a document database. Unlike the relational task spec, data here is denormalised — subtasks live inside the task document as an array, not in a separate collection. This avoids multi-collection joins and suits Firestore's query model. The trade-off: updating a subtask requires a document write to the parent task.

Collection structure

users/{userId}/tasks/{taskId}
  └── Task documents (subtasks as array field)

users/{userId}/recurring_tasks/{recurringTaskId}
  └── Recurring task template

users/{userId}/recurring_completions/{completionId}
  └── One doc per completion event (taskId + date)

users/{userId}/meal_entries/{entryId}
  └── One meal entry per doc

users/{userId}/spend_entries/{entryId}
  └── One spend entry per doc

users/{userId}/spend_categories/{categoryId}
  └── Category + budget config

users/{userId}/chat_threads/{threadId}

Firestore Security Rules (outline)

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // All user data lives under /users/{userId}/
    // Only the authenticated user can read/write their own documents
    match /users/{userId}/{document=**} {
      allow read, write: if request.auth != null
                         && request.auth.uid == userId;
    }
  }
}
07 API & Cloud Functions

All server-side logic lives in Firebase Cloud Functions. The client never calls the Anthropic API directly — all AI calls are proxied through functions. The client calls Firestore directly for reads and simple CRUD writes; Cloud Functions handle AI interactions and any multi-step writes.

Cloud Functions

FunctionTriggerResponsibility
chatTurn HTTPS callable Receives user message + thread ID. Builds full context snapshot from Firestore. Calls Anthropic API. Parses response for proposed actions. Persists both messages. Returns assistant message + proposed actions.
executeAction HTTPS callable Receives a confirmed action payload (type + data). Validates and writes to Firestore. Returns updated document. Called after user confirms an action card.
inferMacros HTTPS callable Receives free-text meal description. Calls Anthropic API with macro inference prompt. Returns structured { cal, protein_g, carbs_g, fat_g, meal_type } for the confirmation card.
taskReviewTurn HTTPS callable Same as chatTurn but context is scoped to a single task. Used by the Task Drawer per-task chat.
seedRecurringToday Cloud Scheduler (daily 00:01) For each active user, checks which recurring tasks don't yet have a completion document for today. No-op if already exists. Ensures the Recurring screen is always current.

Anthropic prompt structure

// System prompt sent on every chatTurn call

You are a calm, warm assistant integrated into a personal life-management app.
The user tracks tasks, meals, and spending through conversation with you.

CAPABILITIES:
- Add, update, or complete tasks
- Log meals (you will estimate calories and macros from free text)
- Record spending against categories
- Answer questions about their data using the context below

RULES:
- Never execute data changes directly. Always propose an action and wait for user confirmation.
- For meals, always return structured macro estimates even if uncertain. Say so if uncertain.
- Identify intent from natural language — the user will not use commands.
- If a message has no data action, respond conversationally.
- Keep responses concise. This is a mobile app.

RESPONSE FORMAT when proposing an action:
Return a JSON block in your response with this structure:
{
  "proposed_actions": [
    {
      "type": "create_task | update_task | complete_task | log_meal | log_spend",
      "payload": { ... },
      "summary": "Human-readable description of the action"
    }
  ]
}

Deep link handling on mobile: The magic link must open the app, not the browser. Configure actionCodeSettings.url to a Firebase Dynamic Link (or your app's custom scheme). 09 Claude Integration Detail

Macro inference flow

When a user types a meal description in chat (or the quick-add on the Meals screen), the flow is:

  • Client calls inferMacros Cloud Function with the description string
  • Function sends to Anthropic with a focused prompt: estimate cal, protein_g, carbs_g, fat_g, and infer meal_type. Return JSON only.
  • Function parses the response and returns the structured object
  • Client renders a confirmation card with the estimates pre-filled in editable fields
  • User adjusts if needed, then confirms → executeAction writes the meal_entry

Macro inference prompt

You are a nutritionist estimating macros for a meal described in natural language.
The user is Indian — assume Indian cooking methods and portion sizes unless stated otherwise.

Return ONLY valid JSON, no preamble, no explanation:
{
  "meal_type": "breakfast | lunch | dinner | snack",
  "cal": number,
  "protein_g": number,
  "carbs_g": number,
  "fat_g": number,
  "confidence": "high | medium | low",
  "note": "brief note if low confidence or ambiguous"
}

User input: "{description}"

Intent classification

The main chat uses a single Anthropic call per turn — no separate classification step. Claude is instructed to identify intent and propose structured actions within the same response. The chatTurn Cloud Function parses the response for a JSON block containing proposed_actions; if none is found, the message is treated as a conversational reply only.

Context window sizing

The context snapshot is built server-side and estimated before sending. For a user with 30 tasks, 5 meal entries, and 8 spend entries, the snapshot is approximately 2,000–3,000 tokens. The chat history for the current thread is included up to the last 20 messages. Total context per turn stays well within the 200K token limit at typical usage.
